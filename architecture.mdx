Fluxora is built using a modern microservices architecture that provides scalability, maintainability, and fault isolation. This guide explains the system's architectural components and how they work together.

## System Architecture

Fluxora consists of the following core components:

- **Service Discovery** (Eureka Server): Enables dynamic service registration and discovery
- **API Gateway**: Single entry point for all client requests with routing and CORS handling
- **Business Microservices**: Four independent services managing different domains
- **Frontend Application**: Next.js-based web interface
- **Database Layer**: PostgreSQL instances with one database per service

## Core Components

### Eureka Server (Service Discovery)

<Card title="Eureka Server" icon="compass">
  **Port**: 8761  
  **Purpose**: Service registration and discovery  
  **Location**: `Microservicios/microservice-eureka`
</Card>

The Eureka Server provides service discovery capabilities, allowing microservices to find and communicate with each other without hard-coded hostnames or ports.

**Key Configuration** (microservice-eureka/src/main/resources/application.properties:1-12):

```properties
spring.application.name=microservice-eureka
server.port=8761
eureka.instance.hostname=eureka-server
eureka.client.register-with-eureka=false
eureka.client.fetch-registry=false
```

All microservices register themselves with Eureka on startup, enabling dynamic service discovery.

### API Gateway

<Card title="API Gateway" icon="door-open">
  **Port**: 8080  
  **Purpose**: Single entry point, routing, CORS handling  
  **Location**: `Microservicios/microservice-gateway`
</Card>

The API Gateway serves as the single entry point for all client requests. It routes requests to appropriate microservices and handles cross-cutting concerns like CORS.

**Route Configuration** (microservice-gateway/src/main/resources/application.properties:10-32):

```properties
# Usuario Service Route
spring.cloud.gateway.server.webflux.routes[0].uri=http://usuario-service:8081
spring.cloud.gateway.server.webflux.routes[0].predicates[0]=Path=/api/usuarios/**

# Inventario Service Route
spring.cloud.gateway.server.webflux.routes[1].uri=http://inventario-service:8082
spring.cloud.gateway.server.webflux.routes[1].predicates[0]=Path=/api/inventario/**

# Cliente Service Route
spring.cloud.gateway.server.webflux.routes[2].uri=http://cliente-service:8083

# Entrega Service Route
spring.cloud.gateway.server.webflux.routes[3].uri=http://entrega-service:8084
spring.cloud.gateway.server.webflux.routes[3].predicates[0]=Path=/api/entregas/**
```

### Cliente Service (Client Management)

<Card title="Cliente Service" icon="users">

  **Database**: `cliente_db` (port 5433)  
  **Location**: `Microservicios/microservice-cliente`

**Key Features**:
- Client CRUD operations

- MapStruct for DTO mapping
- JWT authentication

```java

    @GetMapping("/api/entregas/rutas/cliente/{idCliente}/ruta")

}
```

**Technology Stack**:
- Spring Boot 3.5.4
- Spring Cloud OpenFeign

- PostgreSQL

## Service Communication Patterns

### Service Discovery with Eureka

All microservices register with Eureka on startup and can discover other services dynamically:

```properties
eureka.client.service-url.defaultZone=http://eureka-server:8761/eureka/
eureka.client.fetch-registry=true
eureka.client.register-with-eureka=true
```

### Feign Client for Inter-Service Communication

Services communicate using Spring Cloud OpenFeign, which integrates with Eureka for service discovery:

```java
@FeignClient(name = "microservice-entrega", configuration = FeignClientInterceptor.class)
public interface EntregaServiceClient {
    // Service methods
}
```

The `FeignClientInterceptor` adds JWT tokens to inter-service requests for authentication.

### API Gateway Routing

All external requests flow through the API Gateway, which:
1. Receives the request from the client
2. Resolves the target service via Eureka
3. Routes the request to the appropriate microservice
4. Returns the response to the client

**Database Configuration** (docker-compose.yml:3-24):

```yaml
db-cliente:
  image: postgres:15-alpine
  environment:
    POSTGRES_USER: fluxora
    POSTGRES_PASSWORD: fluxora123
    POSTGRES_DB: cliente_db
```

## Deployment Architecture

Fluxora supports multiple deployment strategies:

### Local Development (Docker Compose)

Using `docker-compose.yml`, all services run in containers with:
- Bridge networking for inter-service communication
- Persistent volumes for database data
- Health checks for database readiness
- Timezone configuration (America/Santiago)

### Cloud Deployment (Render)

The `render.yaml` configuration deploys services to Render.com:
- Managed PostgreSQL database
- Web services for all microservices
- Environment-based configuration
- Automatic SSL/TLS
- Global CDN

<Tip>
The architecture is cloud-agnostic and can be deployed to AWS, Azure, GCP, or any container orchestration platform like Kubernetes.
</Tip>

## Design Principles

### Single Responsibility
Each microservice handles a specific business domain:
- **Usuario**: Authentication & user management
- **Cliente**: Customer management
- **Inventario**: Inventory & materials
- **Entrega**: Deliveries & routing

### Loose Coupling
Services communicate via REST APIs and service discovery, avoiding direct dependencies.

### Database per Service
Each service owns its data, ensuring data isolation and independent scaling.

### API Gateway Pattern
A single entry point simplifies client communication and enables centralized cross-cutting concerns.

### Service Discovery
Dynamic service registration enables elastic scaling and fault tolerance.
